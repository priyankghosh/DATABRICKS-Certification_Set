A data engineering team possesses two distinct tables: march\_transactions, which aggregates all retail transactions for March, and april\_transactions, encompassing all transactions for April. There are no duplicate records between the tables. To compile a new table, all\_transactions, that merges records from both march\_transactions and april\_transactions without introducing duplicates, which command should be executed?

1.  CREATE TABLE all\_transactions AS
2.  SELECT \* FROM march\_transactions
3.  INNER JOIN SELECT \* FROM april\_transactions;
4.  CREATE TABLE all\_transactions AS
5.  SELECT \* FROM march\_transactions
6.  OUTER JOIN SELECT \* FROM april\_transactions;

**Correct answer**

1.  CREATE TABLE all\_transactions AS
2.  SELECT \* FROM march\_transactions
3.  UNION SELECT \* FROM april\_transactions;
4.  CREATE TABLE all\_transactions AS
5.  SELECT \* FROM march\_transactions
6.  INTERSECT SELECT \* from april\_transactions;
7.  CREATE TABLE all\_transactions AS
8.  SELECT \* FROM march\_transactions
9.  MERGE SELECT \* FROM april\_transactions;

Overall explanation

**The UNION SQL command is specifically designed to combine the results of two or more SELECT statements by default without including duplicate rows.** Given that march\_transactions and april\_transactions do not have duplicate records between them, using UNION to merge these datasets into all\_transactions ensures that all unique records from both months are included in the new table. Unlike INNER JOIN, which combines rows based on a matching condition (and is not applicable here since there's no intention to match records), or OUTER JOIN, which would include all records from both tables but require a join condition, UNION straightforwardly amalgamates the two sets of records. INTERSECT and MERGE are not correct in this context, with INTERSECT finding common records (of which there are none by premise) and MERGE not being a valid SQL operation for combining records in this manner.
